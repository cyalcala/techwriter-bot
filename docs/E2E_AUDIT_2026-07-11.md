## First-hand pre-findings (main session, before workflow results)

Recorded so they can be cross-checked against the workflow output:

1. **[perf, streaming hot path]** `ChatMessages.svelte:68` re-runs
   `formatMarkdown(stripDisclaimers(content))` — a full `marked.parse` +
   `DOMPurify.sanitize` over the entire accumulated message — on every
   token batch (rAF-paced via `TokenBatcher`), and over **every** message
   whenever the array is replaced (`messages = [...messages, ...]`).
   Cost grows quadratically with response length during streaming.
   Candidate fix: cache parsed HTML per message keyed by content, and/or
   incremental block-level parsing (only re-parse the trailing block).
2. **[security, medium]** Kroki/mermaid SVG is sanitized by **regex only**
   (`sanitizeSvg`/`sanitizeHtml` in `renderer-loader.ts:512-525`,
   `kroki-renderer.ts:74-81`) and then injected via `{@html}` /
   `innerHTML` (`ArtifactPanel.svelte:249-253`, `ChatArtifactChip.svelte:48`,
   `renderer-loader.ts:315,500`). Regex HTML sanitization is bypassable
   (entity-encoded `javascript:`, nested tags). DOMPurify is already a
   dependency — use `DOMPurify.sanitize(svg, { USE_PROFILES: { svg: true, svgFilters: true } })`
   for the non-iframe SVG paths. (HTML artifacts already render in a
   `sandbox=""` iframe — that path is fine as defense-in-depth.)
3. **[quality, dead code]** `chunkText` in `embed-pipeline.ts:104` (blind
   500/100 slicing) is superseded by the paragraph-aware `chunkDocument`
   in `rag-client.ts` and is only referenced by a test mock. Candidate
   removal (with the stale "~700-char chunks" comment above `BATCH_SIZE`
   pointing at the real chunker).
4. **[robustness, low]** `TokenBatcher` uses `requestAnimationFrame` —
   in background tabs rAF is throttled/paused, so tokens buffer until the
   tab refocuses. Acceptable UX tradeoff, but worth a `setTimeout`
   fallback cap so background streams still make visible progress.

### Confirmed salvage bugs (from the probe, verified by running it)

`src/lib/json-salvage.ts` `scanObjectsFrom` walked one character at a time
over any non-`{` element while scanning an array. Observed results:

| Input shape (slides/blocks array) | Before | Root cause |
|---|---|---|
| starts with a string containing `]` (`["see [1]", {…}]`) | **returns `[]` — all objects lost** | the `]` inside the string hit the array-terminator check |
| starts with a string containing `{` (`["a { b", {…}]`) | **returns `[]`** | the `{` started a bogus object scan that never balanced |
| starts with a nested array (`[["a","b"], {…}]`) | **returns `[]`** | the nested `]` terminated the scan |
| truncated tail object, complete leads | correct (kept leads) | (already fine) |
| `salvageFirstObjectArray` with an earlier `[{x,y},…]` coord array | **returns the coord objects as slides → empty/garbage slides** | it grabbed the first object-array anywhere |

These require the model to emit non-schema-shaped JSON (a leading non-object in
the slides/blocks array, or a wrapper with an incidental object-array first) —
rarer than clean truncation (which already worked), but a real data-loss path
for malformed deck/document output. **Fixed** (see Fix log slice 1).

## Fix log

### Slice 1 — parsing robustness, diagram-render security, streaming perf (commit pending)

**Parsing / ingestion robustness**
- `src/lib/json-salvage.ts`: added `skipValue()` so the array scanner steps over
  whole non-object elements (strings w/ escapes, nested arrays, primitives)
  instead of one char at a time — fixes the three data-loss cases above.
  Added `arrayHasTextContent()` and an optional `accept` predicate on
  `salvageFirstObjectArray()`; `deck-schema.ts` and `doc-schema.ts` pass it so
  the last-resort scan skips incidental coordinate/number arrays and finds the
  real slides/blocks.
- New regression suite `src/tests/json-salvage-edge.test.ts` (6 tests) pins all
  the above (and confirms the common truncation cases still work). Removed the
  stray probe file.

**Diagram-render security (verified in a real browser, not just node)**
- `src/lib/renderer-loader.ts`: `sanitizeSvg` now routes **pure SVG**
  (graphviz/d2/plantuml, user SVG artifacts, and the primary mermaid path) to
  **DOMPurify's parser** (svg profile) — regex HTML sanitizing is bypassable and
  should never be the only guard on the untrusted SVG-artifact path. SVGs that
  embed `<foreignObject>` (kroki's mermaid fallback) still use a **hardened
  regex scrubber** because DOMPurify strips foreignObject HTML and would blank
  mermaid labels (proven — see `output/audit-2026-07-11/mermaid-labels-verification.txt`).
- The regex scrubber now **decodes HTML entities before the URL check**, closing
  an encoded-`javascript:`/`vbscript:`/`data:` bypass (e.g. `&#106;avascript:`),
  while still allowing `data:image/*`.
- Set mermaid `htmlLabels:false` on the client render so the primary mermaid
  path emits pure SVG and gets strict DOMPurify with labels intact.
- Evidence: `output/audit-2026-07-11/svg-sanitize-verification.txt` — 4/4 cases
  pass in real Chrome (attacks stripped, structure + labels preserved).

**Streaming / perf + cleanup**
- `src/lib/markdown.ts`: added a bounded LRU memo (cap 80) around
  `formatMarkdown` so re-rendering the transcript (stream finalize, session
  import, edits) reuses parsed HTML instead of re-running `marked` + DOMPurify
  over every message each time.
- `src/lib/token-batcher.ts`: added a `setTimeout` fallback alongside `rAF` so
  streamed text still flows in backgrounded tabs (where rAF is paused), and
  throttled flushes to ~100 ms once a response passes 4 KB (each flush re-parses
  the whole message upstream, so frame-rate updates on a long message wasted CPU).
- `src/lib/embed-pipeline.ts`: removed the dead `chunkText()` (superseded by the
  paragraph-aware `chunkDocument` in `rag-client.ts`; only a test mock referenced
  it) and its stale mock in `src/tests/rag-client-tools.test.ts`.

Tests after slice 1: **271 passing / 49 files** (was 265/48; +6 salvage-edge,
−8 removed probe). `npm test` green.

### Slice 2 — diagram/artifact classification: kill json→vega, unify detectors

**Confirmed bug:** `src/lib/artifact-detector.ts` (`detectAllArtifacts`, the
post-stream fallback detector) carried its own `langToType` + `validateArtifact`
that had drifted from the canonical classifier in `src/lib/artifact-types.ts`.
Its `langToType` mapped **every** ` ```json ` fence to `vega`, and its vega
validation only checked `JSON.parse` succeeds — so any JSON block in a message
(API responses, config, examples — the bread and butter of a technical-writing
tool) got pulled out of the prose and rendered as a **broken Vega chart**. The
existing test `artifacts.test.ts` "does not classify ordinary json fences as
Vega charts" only passed because its input was under the 50-char early-return
guard; a real-length message hit the bug. The fallback also could not detect
`deck`/`document` fences at all.

**Fix:** replaced the ~60 lines of divergent duplicate logic in
`artifact-detector.ts` with the canonical `normalizeArtifactType` /
`validateArtifact` / `getDefaultArtifactTitle` / `detectCodeLanguage` from
`artifact-types.ts` (which already disambiguates a bare `json` block into
vega/deck/document/none by inspecting the content). The fallback detector now
stays in lockstep with the streaming tag path. Legacy `webcontainer` still
degrades to `code`. Added 2 regression tests (a long plain-json message stays
inline past the length guard; a json fence that really is a Vega-Lite spec
still promotes to a chart).

Tests after slice 2: `artifacts.test.ts` **26/26**, `json-salvage-edge.test.ts`
green (32/32 across the two touched files). NOTE: the full parallel `npm test`

### Slice 3 — ingestion: fix RAG chunker content-loss on early paragraph cuts

**Confirmed bug:** in `src/lib/rag-client.ts`, `chunkDocument` advanced with
`start = Math.max(end - overlap, start + step)` where `step = chunkSize -
overlap`. When a paragraph boundary forced an early `end` (a short paragraph
immediately followed by a long unbroken one), `start + step` jumped **past**
`end`, so the region `[end, start+step]` — the beginning of the long paragraph —
was never emitted in any chunk and was **silently dropped from the RAG index**
(unretrievable). Traced with a `"Intro."`-then-long-paragraph document: the
first ~570 chars of the long paragraph disappeared.

**Fix:** `start = (end - start >= overlap) ? end - overlap : end`. A normal
full-size chunk still overlaps the previous by `overlap` (unchanged behavior);
an early/short chunk continues from `end` itself, which is contiguous (no gap)
and avoids re-cutting the same boundary (no fragment spam). Removed the now-dead
`step` local. New regression test plants distinct `NEEDLE_*` tokens across the
previously-dropped region and asserts they all survive into chunks, with a
bounded chunk count. Tests: `document-ingest.test.ts` **10/10**, and the RAG
suite (`rag-citations`, `rag-document-registry`, `rag-client-tools`,
`document-ingest`) **21/21**.
